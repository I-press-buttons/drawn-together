# Code Health & Architecture — Design

## Context

"Couple Questions" is a single-page party card game (see `PRODUCT.md`) currently implemented as one 2000-line `index.html` (inline CSS + JS) plus a stdlib-only Python HTTP server (`server.py`) that persists user-created "question packs" to `question_packs.json`. There is no git repository, no tests, and no `.gitignore`. The 108 base questions are duplicated — once as a parsed template string in JS, once in `Couple_Question_Game.txt`.

This spec covers a code-health pass: structural cleanup, backend hardening, and repo setup. It does not change game behavior, UX, or visual design — PRODUCT.md's product/UX direction is out of scope here.

## Goals

- Split the monolithic `index.html` into markup/style/script files, no build tooling.
- Eliminate the duplicated question data (JS template string vs. `Couple_Question_Game.txt`) — one `questions.json` source of truth.
- Harden `server.py` against concurrent-write races and unbounded/invalid input, while staying dependency-free.
- Initialize git, add a `.gitignore`, clean up junk data in `question_packs.json`.
- Add minimal stdlib-based tests for the pack CRUD API.

## Non-goals

- No framework migration (Flask/FastAPI) and no frontend build step (Vite/bundler) — confirmed zero-dependency preference.
- No UX/visual/product changes.
- No deployment/hosting setup.

## Design

### 1. File split

- `index.html`: markup only, `<link rel="stylesheet" href="style.css">`, `<script src="app.js" defer></script>`.
- `style.css`: current inline `<style>` block content, unchanged.
- `app.js`: current inline `<script>` block content, unchanged logic — plain script (no ES modules), since it's a single file and adding module semantics buys nothing here.

`server.py`'s `SimpleHTTPRequestHandler` already serves static files from the working directory, so no server changes are needed to serve the new files.

### 2. Question data consolidation

- New `questions.json` at repo root: array of `{ text, rarity, category }`, generated by parsing the current template string once (values same as today, `category` "General" or "Future Us").
- `app.js` fetches `questions.json` at boot (same pattern as `loadPacks()`), replacing `QUESTIONS` global + `parseQuestions()`.
- Delete `Couple_Question_Game.txt` (fully redundant once `questions.json` exists).
- `server.py` needs no changes to serve `questions.json` (static file).

### 3. Backend hardening (`server.py`)

- Add a module-level `threading.Lock()` guarding every `load_packs()` → mutate → `save_packs()` sequence in `do_POST`/`do_PUT`/`do_DELETE`, preventing lost updates from concurrent requests (e.g. two phones editing packs at once).
- Add a `MAX_BODY_BYTES` cap (e.g. 1 MB) in `read_json_body`; reject oversized bodies with a 413-equivalent JSON error instead of reading unbounded input into memory.
- Add server-side length validation matching the existing HTML `maxlength` attributes (pack name ≤ 60 chars, question text ≤ 300 chars) so the API can't be used to bypass client-side limits.

### 4. Repo & data cleanup

- `git init` at project root.
- `.gitignore`: `.impeccable/`, `__pycache__/`, `*.pyc`, `.DS_Store`.
- Reset `question_packs.json` to `[]` (removes the `"Whatdoyodddddddd"` junk test pack).
- Initial commit capturing the restructured project.

### 5. Tests

- `test_server.py` using stdlib `unittest` + `http.client` (or `urllib`), spinning up `GameHandler` on an ephemeral port per test class.
- Coverage: create pack (success + missing-name 400), toggle pack, delete pack, add question to pack (success + missing-text 400 + oversized-text 400), delete question, 404s for unknown pack/question ids.
- Run via `python3 -m unittest test_server.py` — no pytest dependency.

## Error handling

Existing error responses (400 for missing fields, 404 for unknown ids) are preserved. New error cases added by hardening: 413-style JSON error for oversized request bodies, 400 for over-length name/text fields.

## Out of scope / explicitly deferred

- Path traversal review of `SimpleHTTPRequestHandler`'s static file serving — stdlib's handler already normalizes paths safely; no changes planned unless the plan phase turns up a concrete issue.
- Multi-file JS modularization (splitting `app.js` further by concern) — deferred; current split (HTML/CSS/JS) is the agreed scope for this pass.
